# Taxi Orders — Implementation Spec

A web app for managing taxi orders for a local company (Bolivia), optimized for fast delivery to the customer.

**Stack:** Next.js (App Router) · React · TypeScript · Prisma + PostgreSQL (Neon) · NextAuth (Auth.js) · Tailwind + shadcn/ui · Zod · SWR (polling) · Server Actions for mutations. Deployed on Vercel.

---

## 1. Decisions locked

### Data & integrity
- **Timezone:** `America/La_Paz` (UTC−4, no DST). All day boundaries — the code counter and history windows — resolve against this, not UTC.
- **Order code `P-MMDD-NNNN`** (e.g. `P-0612-0042`): generated by an **atomic daily counter**. A `DailyCounter` table keyed by the local date is incremented in the **same transaction** as the order insert. Gapless, sequential, collision-free under concurrency.
- **Status state machine:** `WAITING → DISPATCHED` and `WAITING → ARCHIVED`, both **reversible** to `WAITING`. Reverting nulls `dispatchedAt`/`archivedAt`; `createdAt` is unchanged, so a reverted order resurfaces at its true FIFO position (typically near the top, red badge).
- **Indexes:** `@@index([status, createdAt])` for the FIFO inbox + `@@index([createdAt, id])` for the history range scan and cursor pagination.
- **Anti-abuse on the public form:** hidden honeypot field + minimum fill-time check + DB-backed per-IP throttle (`ipHash` column; reject if the same hashed IP exceeds a threshold within a recent window). No external service.

### Visitor (anonymous)
- **Scope = the form only.** No login. No public status-lookup page — the code is a reference number the customer quotes over WhatsApp. (This avoids enumeration of sequential codes.)
- **Required:** `passengerName`, `phone`, `pickupAddress`. **Optional:** `destination`, `notes`.
- **Phone validation:** strict Bolivian mobile — strip spaces/dashes, require exactly 8 digits starting with `6` or `7`, store normalized E.164 `+591XXXXXXXX`. Same Zod schema runs on client and server.
- **On submit (Server Action):** persist the order, **return the code**. Success message:
  > `Pedido recibido. Su código es {code}. Espere confirmación por WhatsApp; puede consultar por su pedido con este código.`
  On error, show an error message. (Real WhatsApp integration is out of scope.)

### Operator (authenticated)
- **Auth:** NextAuth/Auth.js **Credentials** provider, argon2-hashed passwords, **JWT** session strategy (no DB session adapter). Single operator role.
- **Provisioning:** manual insert in Neon, using the `npm run hash-pw "<password>"` helper to produce a valid `passwordHash`. No public sign-up, no admin UI.
- **Protection:** operator routes and all read route handlers are guarded by the session.

### Inbox (status WAITING)
- Load the **entire** WAITING queue, **FIFO ascending** (oldest at top), **no pagination**. The operator sees the whole queue.
- One **shadcn Card** per order showing all fields in plain text: code, name, phone, origin, destination, notes.
- **Waiting badge** showing text + color together (e.g. `esperando 27 min`), recomputed each render:
  - 🟢 green: 0–15 min
  - 🟡 amber: 15–25 min
  - 🔴 red: >25 min
  - Thresholds are configurable constants. Purely a visual priority signal — no automatic action, no auto-archive.
- **Actions per card** (no detail view):
  1. **Mark dispatched** (Server Action)
  2. **Archive** (Server Action — always manual)
  3. **Copy driver data block** (client component, `navigator.clipboard`)
  4. **Copy customer message** (client component, `navigator.clipboard`)
- **Action UX:** await the Server Action, then trigger SWR revalidation. Buttons enter a disabled/pending state during the round-trip to prevent double-firing.

#### Clipboard templates
Empty optional fields are omitted automatically.

**Driver data block** (with company-name header):


### History
- **Read-only table** + one guarded "return to waiting" action per row (the only mutation here).
- **Columns:** code, time (`createdAt`), status (waiting/dispatched/archived).
- **Pagination:** cursor/keyset on `(createdAt, id)` — stable under inserts, performant at any depth. Next/Prev navigation.
- **Range selector** + optional status filter (default: all statuses). Never load everything.

### View updates (polling)
- Inbox and history use **SWR** with a **~4000ms** refresh interval, each with its own **read route handler** as the fetcher.
- SWR **pauses polling when the tab is hidden** (useful for an operator switching between WhatsApp and the app).
- **No** WebSockets, **no** SSE, **no** real-time. No cron, no scheduler, no auto-archive.

### Deploy / infra
- **Vercel** (Node runtime, no cron).
- **Neon:** pooled `DATABASE_URL` (PgBouncer) for runtime + unpooled `DIRECT_URL` for migrations/introspection.
- **`AUTH_SECRET`** env for NextAuth.
- **`/api/health`** endpoint for an external uptime monitor.
- Neon backups enabled.

## 2. Assumptions (running with these unless changed)

1. **Company name:** `{NOMBRE_EMPRESA}` placeholder until the real name is provided. Stored as a single config constant so it can be set in one place.
2. **History default window on load:** last 7 days, with a one-click "hoy" (today) toggle.
3. **Uptime monitor:** a free external pinger (UptimeRobot or Better Stack) hitting `/api/health`. This is observability, not a core dependency, so it doesn't violate "no external services" in spirit.
4. **Single operator role** — no admin tier, no in-app user management.
5. **Throttle window/threshold** for the per-IP limit are configurable constants (initial guess: N orders per 10 min per IP) — tune after observing real traffic.

## 3. Data model (Prisma)

```prisma
enum Status {
  WAITING
  DISPATCHED
  ARCHIVED
}

model Order {
  id            String    @id @default(cuid())
  code          String    @unique            // P-MMDD-NNNN, generated in create logic
  passengerName String
  phone         String                        // normalized +591XXXXXXXX
  pickupAddress String
  destination   String?
  notes         String?
  status        Status    @default(WAITING)
  dispatchedAt  DateTime?
  archivedAt    DateTime?
  ipHash        String?                        // anti-abuse throttle
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt

  @@index([status, createdAt])                 // FIFO inbox
  @@index([createdAt, id])                      // history range + cursor
}

model DailyCounter {
  day   String @id          // local-date key, e.g. "2026-06-12" (America/La_Paz)
  count Int    @default(0)
}

model User {
  id           String   @id @default(cuid())
  email        String   @unique
  passwordHash String
  name         String?
  createdAt    DateTime @default(now())
}
```

**Code generation (inside a transaction):**
1. Compute local date key in `America/La_Paz`.
2. `upsert` + atomic `increment` on `DailyCounter` for that key → `n`.
3. Format `P-{MMDD}-{n.padStart(4,"0")}`.
4. Insert the order with that code. All in one transaction so the counter and the order are consistent.

## 7. Implementation notes (decisions made while building)

These reflect what actually shipped and supersede assumptions in the spec where they differ.

### Stack versions / Prisma 7
- Scaffolded with **Next.js 16** (App Router, Turbopack), **React 19**, **Tailwind 4**, **Zod 4**, **pnpm**.
- **Auth (Phase 3):** **Auth.js v5** (`next-auth@5.0.0-beta`) — Credentials provider, `session.strategy: "jwt"`, `trustHost: true`. Passwords hashed with **argon2id** (`argon2` package; `argon2` is in `pnpm-workspace.yaml › onlyBuiltDependencies` so its native binding builds). `/inbox` and `/login` are guarded at the **page level** via `await auth()` + `redirect()` (no middleware → keeps argon2 off the edge runtime). `AUTH_SECRET` is set in `.env`.
- **Prisma 7** (newer than the spec assumed) changed two things we had to adapt to:
  - Connection URLs are **no longer in `schema.prisma`** — they live in `prisma.config.ts` (loads `.env` via `dotenv`; uses `DIRECT_URL` → `DATABASE_URL` for Migrate/Studio).
  - The runtime client requires a **driver adapter**. We use **`@prisma/adapter-pg`** (node-postgres) — plain TCP, so the *same* client works against local Docker Postgres (dev) and Neon's pooled endpoint (prod). No code swap at deploy. Singleton in `src/lib/db.ts`.
  - Generated client lives at **`src/generated/prisma`** (gitignored); import from `@/generated/prisma/client`. Regenerated on `postinstall` / `build`.

### Local development
- **`docker-compose.yml`** runs Postgres 17 on `localhost:5432` (user/pass/db all `taxi`). `.env` points at it; **`.env.example`** documents the Neon format. Swap to Neon at Phase 7.

### Operator provisioning
- Canonical: `pnpm hash-pw "<password>"` → paste the hash into a `User` row in Neon.
- Convenience (local / first seed): `pnpm create-operator "<email>" "<password>" ["<name>"]` (idempotent upsert).
- For local testing, seed a dev operator with your own password via `pnpm create-operator` (Docker DB only — never reuse a prod password).

### Open items / follow-ups
